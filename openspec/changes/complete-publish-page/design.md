## Context

发布页面是拼车小程序的核心功能，需要在现有的空白页面基础上实现完整的发布流程。项目使用 uni-app + Vue 3 + TypeScript 技术栈，已有基础的 API 层和类型定义。

**当前状态：**
- `src/pages/publish/index.vue` 是空白页面
- `src/api/ride.ts` 已有基础 API 结构但缺少发布接口
- `src/types/index.ts` 已定义 Ride 和 RideType 类型
- 后端 API 暂时使用 mock 数据

**约束：**
- 必须使用 uni-app 组件（兼容微信小程序）
- 表单需要良好的用户体验和错误提示
- 暂不集成地图服务，先实现手动输入地址

## Goals / Non-Goals

**Goals:**
- 实现完整的发布表单，包含所有必要字段
- 支持两种发布类型切换（找车/找乘客）
- 实现客户端表单验证
- 调用后端 API 提交数据
- 提供清晰的成功/失败反馈
- 发布成功后跳转到首页

**Non-Goals:**
- 不实现地图选点功能（后续优化）
- 不实现草稿保存功能
- 不实现图片上传功能
- 不实现支付相关功能

## Decisions

### 1. 表单状态管理

**决策：** 使用 Vue 3 Composition API 的 `reactive` 管理表单数据

**理由：**
- `reactive` 适合管理对象形式的表单数据
- 代码更简洁，易于维护
- 符合 Vue 3 最佳实践

**替代方案：**
- 使用多个 `ref`：会导致代码冗余，每个字段都需要单独声明
- 使用 Pinia：对于单页面表单过于重量级

### 2. 表单验证策略

**决策：** 实现两层验证：客户端实时验证 + 提交前完整验证

**理由：**
- 实时验证提供即时反馈，改善用户体验
- 提交前验证确保数据完整性
- 客户端验证减少不必要的 API 调用

**验证规则：**
- 出发地/目的地：必填，长度 2-50 字符
- 出发时间：必填，不能早于当前时间
- 座位数：找乘客模式必填，1-8 人
- 价格：可选，0-9999 元
- 备注：可选，最多 200 字符

### 3. 拼车类型切换

**决策：** 使用 uni-app 的 `<radio-group>` 组件实现类型选择

**理由：**
- 原生组件，性能好
- 符合微信小程序设计规范
- 支持样式自定义

**交互逻辑：**
- 切换类型时，座位数字段显示/隐藏
- 找车模式：隐藏座位数
- 找乘客模式：显示座位数（必填）

### 4. 时间选择

**决策：** 使用 uni-app 的 `<picker mode="datetime">` 组件

**理由：**
- 原生组件，体验一致
- 自动处理时区和格式
- 支持最小时间限制（当前时间）

**格式：** YYYY-MM-DD HH:mm

### 5. 地点输入

**决策：** 第一版使用 `<input>` 手动输入，预留地图选点接口

**理由：**
- 快速实现，降低复杂度
- 地图服务需要额外配置和权限
- 手动输入可满足基本需求

**后续优化：**
- 集成腾讯地图或高德地图
- 实现地点搜索和选点
- 显示地点详情和距离

### 6. API 设计

**决策：** 在 `src/api/ride.ts` 中添加 `publishRide` 方法

**接口定义：**
```typescript
interface PublishRideParams {
  type: RideType
  departure: string
  destination: string
  departureTime: string
  seats?: number
  price?: number
  note?: string
}

function publishRide(params: PublishRideParams): Promise<Ride>
```

**错误处理：**
- 网络错误：显示 toast 提示重试
- 验证错误：显示具体字段错误
- 服务器错误：显示通用错误信息

### 7. 数据流

**流程：**
1. 用户填写表单
2. 实时验证（失焦时）
3. 点击发布按钮
4. 完整验证
5. 调用 API
6. 显示 loading
7. 成功：toast 提示 + 跳转首页
8. 失败：toast 提示 + 保持页面

## Risks / Trade-offs

### 风险 1：手动输入地址可能不准确
**影响：** 用户可能输入错误或不完整的地址，影响匹配效果

**缓解措施：**
- 提供输入示例（placeholder）
- 后续版本集成地图选点
- 在首页显示时提供地址编辑功能

### 风险 2：后端 API 暂时是 mock 数据
**影响：** 发布的数据不会真正保存，刷新后消失

**缓解措施：**
- 在 API 层预留真实接口的实现
- 使用 localStorage 临时存储发布的数据
- 后端就绪后快速切换

### 风险 3：表单字段较多，可能影响用户体验
**影响：** 用户可能觉得填写繁琐，放弃发布

**缓解措施：**
- 优化表单布局，分组显示
- 提供合理的默认值
- 标注必填和可选字段
- 后续考虑分步骤填写

### Trade-off：简化实现 vs 完整功能
**选择：** 优先实现核心功能，地图选点等高级功能后续迭代

**理由：**
- 快速验证核心流程
- 降低首次开发复杂度
- 根据用户反馈优化

## Migration Plan

**部署步骤：**
1. 实现发布页面 UI 和逻辑
2. 添加 API 接口（mock 实现）
3. 在微信开发者工具中测试
4. 真机测试表单交互和验证
5. 发布到测试环境

**回滚策略：**
- 如果发现严重问题，可以临时隐藏发布按钮
- 保持页面路由配置，避免影响其他页面
