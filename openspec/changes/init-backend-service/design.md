## Context

微信拼车小程序当前只有前端实现,所有数据都是模拟数据。需要构建一个完整的后端服务来支撑业务功能。

**当前状态:**
- 前端: uni-app + Vue3 + TypeScript
- 后端: 无
- 数据: 前端模拟数据

**技术约束:**
- 必须支持微信小程序的用户登录体系
- 需要实时聊天功能
- 预算有限,优先使用免费/低成本方案
- 团队熟悉 Node.js 和 TypeScript

**利益相关方:**
- 前端开发者: 需要稳定的 API 接口
- 用户: 需要快速响应和稳定的服务
- 运维: 需要易于部署和维护的架构

## Goals / Non-Goals

**Goals:**
- 构建可扩展的 RESTful API 服务
- 实现用户认证和授权机制
- 提供拼车信息的 CRUD 操作
- 集成腾讯云 IM 实现实时聊天
- 设计清晰的数据库模型
- 提供完整的 API 文档

**Non-Goals:**
- 不实现复杂的推荐算法(后期优化)
- 不实现支付功能(MVP 阶段)
- 不实现地图路线规划(使用第三方 API)
- 不实现管理后台(后期开发)

## Decisions

### 1. 后端框架: Nest.js

**决策:** 使用 Nest.js 作为后端框架

**理由:**
- **TypeScript 原生支持**: 与前端技术栈一致,代码可复用
- **模块化架构**: 清晰的分层结构,易于维护和扩展
- **依赖注入**: 便于测试和解耦
- **生态完善**: 内置支持 TypeORM、Swagger、JWT 等
- **企业级**: 适合中大型项目,有完整的最佳实践

**备选方案:**
- Express.js: 太轻量,需要自己搭建架构
- Koa.js: 生态不如 Nest.js 完善
- Fastify: 性能更好,但生态和文档不如 Nest.js

### 2. 数据库: MySQL + TypeORM

**决策:** 使用 MySQL 8.0 + TypeORM

**理由:**
- **MySQL**: 成熟稳定,社区支持好,云服务商都支持
- **TypeORM**: Nest.js 官方推荐,TypeScript 支持好,Active Record 模式易用
- **关系型数据库**: 拼车业务有明确的关系模型(用户-订单-拼车信息)

**备选方案:**
- PostgreSQL: 功能更强大,但团队不熟悉
- MongoDB: 不适合关系型数据
- Prisma: 更现代,但生态不如 TypeORM 成熟

### 3. 即时通讯: 腾讯云 IM

**决策:** 使用腾讯云 IM SDK

**理由:**
- **免费体验版**: 100 个 DAU 免费,适合 MVP 阶段
- **微信生态**: 与微信小程序集成好
- **功能完善**: 支持单聊、群聊、消息存储、离线推送
- **稳定可靠**: 腾讯自家产品,稳定性有保障
- **开发成本低**: SDK 封装好,无需自建 WebSocket 服务

**备选方案:**
- 自建 Socket.io + Redis: 开发成本高,需要处理消息存储、离线推送等
- 环信/融云: 功能类似,但与微信生态集成不如腾讯云

### 4. 认证方案: JWT + 微信登录

**决策:** 使用 JWT token + 微信小程序登录

**理由:**
- **无状态**: JWT 适合分布式部署
- **微信登录**: 使用微信 code 换取 openid,无需用户输入密码
- **简单高效**: 前端存储 token,后端验证 token

**流程:**
1. 小程序调用 `wx.login()` 获取 code
2. 前端将 code 发送到后端
3. 后端用 code 换取 openid 和 session_key
4. 后端生成 JWT token 返回给前端
5. 前端后续请求携带 token

### 5. 项目结构

**决策:** 采用模块化分层架构

```
backend/
├── src/
│   ├── modules/
│   │   ├── auth/          # 认证模块
│   │   ├── user/          # 用户模块
│   │   ├── ride/          # 拼车模块
│   │   ├── chat/          # 聊天模块(腾讯云 IM 集成)
│   │   └── common/        # 公共模块
│   ├── config/            # 配置
│   ├── database/          # 数据库相关
│   │   ├── entities/      # 实体定义
│   │   └── migrations/    # 数据库迁移
│   ├── guards/            # 守卫(权限控制)
│   ├── interceptors/      # 拦截器
│   ├── filters/           # 异常过滤器
│   └── main.ts            # 入口文件
├── test/                  # 测试
├── .env.example           # 环境变量示例
└── package.json
```

**理由:**
- **模块化**: 每个业务模块独立,职责清晰
- **分层**: Controller → Service → Repository,符合 Nest.js 最佳实践
- **可扩展**: 新增功能只需添加新模块

### 6. API 设计规范

**决策:** RESTful API + 统一响应格式

**响应格式:**
```typescript
{
  code: number,      // 业务状态码
  message: string,   // 提示信息
  data: any         // 数据
}
```

**错误处理:**
- 使用 HTTP 状态码 + 业务错误码
- 统一异常过滤器处理所有错误
- 提供清晰的错误信息

## Risks / Trade-offs

### 1. 腾讯云 IM 免费版限制
**风险:** 免费版只支持 100 DAU,超出需要付费

**缓解措施:**
- MVP 阶段足够使用
- 提前规划付费方案
- 设计时保持 IM 模块独立,便于后期切换

### 2. 数据库性能
**风险:** MySQL 在高并发下可能成为瓶颈

**缓解措施:**
- 合理设计索引
- 使用 Redis 缓存热点数据
- 后期可以考虑读写分离

### 3. 微信登录依赖
**风险:** 依赖微信登录,无法支持其他平台

**缓解措施:**
- 设计时抽象认证接口
- 预留其他登录方式的扩展点

### 4. TypeORM 性能
**风险:** TypeORM 在复杂查询时性能不如原生 SQL

**缓解措施:**
- 复杂查询使用 QueryBuilder 或原生 SQL
- 关键接口做性能测试和优化

- 使用 Git 版本控制,可快速回退代码

## Open Questions

1. **服务器部署方案**: 使用云服务器还是 Serverless?
   - 建议: 初期使用云服务器(如腾讯云轻量应用服务器),成本可控

2. **图片存储**: 用户头像、拼车图片存储在哪里?
   - 建议: 使用腾讯云 COS 对象存储,与 IM 服务同一厂商

3. **日志和监控**: 如何收集日志和监控服务状态?
   - 建议: 使用 Winston 记录日志,后期接入监控平台

4. **API 限流**: 如何防止接口被恶意调用?
   - 建议: 使用 Nest.js 的 ThrottlerModule 实现限流

5. **数据备份**: 数据库如何备份?
   - 建议: 使用云服务商的自动备份功能,每天备份一次
